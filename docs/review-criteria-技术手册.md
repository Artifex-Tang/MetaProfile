# 技术手册审查标准（提示词）

> 可复用：审任何前端/技术手册时，把下方"审查任务"段作为提示词喂给审查者（人或 LLM），再附被审文档路径。

## 审查任务（Prompt）

```
你是技术文档评审。仅依据被审文档与可访问的仓库实际文件，逐条给出问题。每条 finding 形如：
[严重度] 章节/位置 — 问题（引用原文）— 修正建议。
严重度三级：阻断（事实错误/无法复现）、一般（不完整/不一致/有歧义）、建议（文风/可读性）。
不要凑数；无问题就明说。最后给结论：PASS / 需修订（列出阻断项数）。
```

## 审查标准（7 维）

1. **准确性 Accuracy**
   - 命令、版本号、文件路径、目录结构、用例数、导出符号名与仓库实际一致。
   - 核对方法：对照 `package.json`、`vite.config.ts`、被测源文件、测试文件。

2. **可复现性 Reproducibility**
   - 读者照手册能跑通 `npm test`。
   - 依赖安装、setup 文件、脚本配置、运行命令完整无遗漏。
   - 环境/前置条件（Node 版本、registry）若有影响需点明。

3. **完整性 Completeness**
   - 覆盖：选型理由、如何运行、目录结构、覆盖范围（用例表）、不纳入项+原因+替代、测试规约、mock 范式、如何新增测试、CI 建议、已知告警、与设计文档/需求的对应关系。
   - 凡手册主题（前端测试）相关关键点不得缺失。

4. **一致性 Consistency**
   - 与项目既有约定一致（中文、文风、不使用第一人称）。
   - 与代码一致：文件名、导出名、函数签名、用例计数等。
   - 手册内部前后一致：同一数字/术语/路径不出现矛盾。

5. **可操作性 Actionability**
   - "如何新增测试""mock 范式"等步骤具体、可照抄，非泛泛而谈。
   - 代码片段可直接复制使用。

6. **文风 Style**
   - 平实、紧凑、信息密度高；一句话一个事实，给依据不给评价。
   - 禁套话/空话/营销腔（"赋能/打造/助力/大幅/极大地"）；禁无数据的程度副词；禁段尾空泛总结。
   - 每句有实指（具体命令/数值/路径/符号）。

7. **边界声明 Boundary**
   - 明确**不**纳入单测的部分 + 原因 + 替代方案（手动/集成/E2E/后端测）。
   - 不留"看起来全覆盖、实则有缺口"的印象。
