| 项 | 内容 |
|---|---|
| 适用范围 | `frontend/`（React + Vite + TypeScript） |
| 框架 | Vitest 4 · @testing-library/react 16 · @testing-library/jest-dom 6 · jsdom 29 |

| 状态 | 基建就绪 · 26 用例全绿 |

---

## 1. 选型说明

| 选择 | 理由 |
|---|---|
| Vitest | 与 Vite 同构，复用 `vite.config.ts`，零 Babel/Jest 配置；ESM/TS 原生支持 |
| @testing-library/react | 以用户视角断言 DOM（角色/文本），不耦合实现细节 |
| @testing-library/jest-dom | 扩展匹配器（`toBeInTheDocument` 等），可读性高 |
| jsdom | Node 端 DOM 模拟，免浏览器即可跑单测 |
| 不写快照测试 | 快照易腐化、失败语义弱；改用显式断言 |

## 2. 如何运行

```bash
cd frontend

# 跑全部（单次，CI 用）
npm test

# 监听模式（开发时）
npm run test:watch

# 跑单个文件
npx vitest run src/utils/crossProfile.test.ts

# 按用例名过滤
npx vitest run -t "可跳类型"
```

`npm test` 等价于 `vitest run`，退出码非 0 即有失败。

## 5. 不纳入 jsdom 单测的部分（及替代方案）

| 不测 | 原因 | 替代 |
|---|---|---|
| `RelationGraph.tsx`（G6 渲染/点击） | `@antv/g6` 依赖 Canvas/真实布局 API，jsdom 无 canvas，渲染与 `node:click` 事件链无法可靠模拟 | 手动冒烟 + 后续 E2E（Playwright，见 §7）。纯逻辑（`relLabel`、`isNavType`）已抽出单测 |
| 含图谱的 `Profile*` 页面（Org/Person/Project/Tech） | 依赖 G6 + Drawer + 后端接口 + 路由，单测成本高、收益低 | 集成/E2E；后端契约由 `pytest` 覆盖。`ProfileGeneric` 同理不纳入 |
| 后端 API 调用 | 属后端范畴 | 后端已有 pytest；前端如需可在集成测用 MSW 拦截 |

原则：**纯逻辑与可隔离组件入单测；涉及画布/网络/路由栈的入集成或手动**。

## 6. 测试规约

1. **不使用全局 API**：`globals: false`，每个测试文件显式 `import { describe, it, expect, vi } from 'vitest'`，避免隐式依赖。
2. **断言用 jest-dom 匹配器**：`toBeInTheDocument()`、`toHaveTextContent()` 等，优于裸 `expect(node).toBeTruthy()`。
3. **纯函数优先**：能从组件抽出的判断/解析逻辑，抽到 `utils/` 并单测（如 `isNavType`、`parseFromQuery`）。
4. **Hook 用 `renderHook`**：配 `MemoryRouter` 等 Provider 作 `wrapper`。
5. **mock react-router-dom 的标准范式**（见 `useCrossProfileJump.test.tsx`）：
   ```ts
   const { navigate } = vi.hoisted(() => ({ navigate: vi.fn() }))
   vi.mock('react-router-dom', async (importOriginal) => {
     const actual = await importOriginal<typeof import('react-router-dom')>()
     return { ...actual, useNavigate: () => navigate }   // 仅覆盖 useNavigate，保留 useLocation 真实
   })
   ```
   - `vi.hoisted` 解决 `vi.mock` 提升至顶层时的变量引用问题；
   - `importOriginal` 保留其余导出（如 `useLocation`/`MemoryRouter`），使路由上下文真实可用。
6. **清理自动**：`src/test/setup.ts` 中 `afterEach(cleanup)`，无需各文件手写。
7. **用例自解释**：中文用例名描述行为与边界（"不可跳类型（enterprise）不触发 navigate"）。

## 7. 如何新增测试

以给 `utils/crossProfile.ts` 新增函数 `foo()` 为例：

1. 在 `src/utils/crossProfile.test.ts` 增 `describe('foo', () => { it('…', () => { expect(foo(input)).toBe(expected) }) })`。
2. `npm run test:watch` 自动重跑，观察绿/红。
3. 涉及组件：新建 `Foo.test.tsx`，`import { render, screen } from '@testing-library/react'`，断言 DOM。
4. 涉及路由：套 `MemoryRouter` wrapper；需断言跳转则按 §6.5 mock `useNavigate`。

## 8. CI 建议

- 流水线在 `build` 前加一步 `cd frontend && npm ci && npm test`。
- Vitest 默认退出码即可作通过判定，无需额外解析。
- 可启用 `--coverage`（需装 `@vitest/coverage-v8`）输出覆盖率门禁，当前未引入以避免额外依赖。

## 9. 已知告警（无害）

运行时可能输出：

```
[vite] warning: `esbuild` option was specified by "vite:react-babel" plugin. This option is deprecated, please use `oxc` instead.
```

为 Vitest 4 与 `@vitejs/plugin-react` 在 Vite 5 下的兼容性提示，不影响用例结果。升级至 Vite 6/Rolldown 后可消除。

## 10. 与设计文档的对应

测试覆盖 `design-跨画像跳转-方案A.md` §10「测试要点」中可在 jsdom 验证的部分：

| 设计文档要点 | 对应用例 |
|---|---|
| 仅四类画像可跳（大小写归一） | `isNavType` 全组、hook「大写 type 归一」 |
| enterprise/strategy 不可跳 | `isNavType` 扩展类型、hook「不可跳类型不触发」 |
| 跳转上下文（含 evidence）完整携带 | hook「路径与 state 正确」 |
| `?from=` 兜底还原来源（不还原关系） | `parseFromQuery` 全组、hook「ctx 兜底」 |
| 面包屑来源可点击返回、刷新/分享退化 | `JumpBreadcrumb` 点击返回 + null 不渲染 |
| 悬空目标 404 / 拖拽误触发 | 需真实后端与浏览器 → 手动冒烟（见设计文档 §10） |
