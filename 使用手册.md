## 3. 首次部署

### 3.1 进入部署目录

```bash
cd MetaProfile/deploy
```

### 3.2 配置环境变量

```bash
# 复制示例文件
cp .env.example .env
```

**最小化配置**（不使用 LLM 功能）：`.env` 默认值已可用，无需修改。

**启用 LLM 功能**（可选）：编辑 `.env`，填入真实 API Key：

```env
LLM_PROXY_API_KEY=sk-your-master-key          # LiteLLM 主密钥（自定义）
QWEN_API_KEY=sk-xxxx                           # 阿里云 DashScope
DEEPSEEK_API_KEY=sk-xxxx                       # DeepSeek
```

### 3.3 构建并启动

```bash
docker compose up -d --build
```

首次执行会拉取并构建所有镜像，约需 **5~10 分钟**（视网络速度）。

完成后应看到：

```
Container metaprofile-frontend-1        Started
Container metaprofile-topic_selection-1 Started
...
```

### 3.4 确认所有服务健康

```bash
docker compose ps
```

所有容器 `STATUS` 列应显示 `(healthy)`：

```
NAME                            STATUS
metaprofile-elasticsearch-1     Up X minutes (healthy)
metaprofile-frontend-1          Up X minutes (healthy)
metaprofile-postgres-1          Up X minutes (healthy)
metaprofile-redis-1             Up X minutes (healthy)
metaprofile-profile_tech-1      Up X minutes (healthy)
...（共 11 个，全量模式 14 个）
```

> **注意**：elasticsearch 启动较慢（约 60 秒），其他服务等待其 healthy 后才会启动。

### 3.5 打开前端

浏览器访问：**http://localhost**

## 4. 启动与停止

### 日常启动（已构建过）

```bash
cd MetaProfile/deploy
docker compose up -d
```

### 停止（保留数据）

```bash
docker compose down
```

### 停止并清除所有数据

```bash
docker compose down -v   # 警告：删除 postgres / ES / neo4j 数据卷，不可恢复
```

### 重新构建（代码有变更后）

```bash
docker compose up -d --build
```

### 查看日志

```bash
# 所有服务
docker compose logs -f

# 单个服务
docker compose logs -f profile_tech
docker compose logs -f frontend
```

### 重启单个服务

```bash
docker compose restart profile_tech
```

## 5. 前端界面操作

访问 **http://localhost**，左侧导航栏包含 8 个页面。

### 5.1 总览（Dashboard）

首页仪表盘，展示：
- 4 类画像总数统计卡片
- 前沿技术 Top 5（按融合评分排序）
- 最新告警列表

### 5.2 技术画像

**搜索**：在搜索框输入关键词，回车触发全文搜索。

**新建**：点击「新建」按钮，填写中文名、英文名、领域、状态、摘要，提交。

**批量导入**：点击「批量导入」，选择 JSON 文件（格式见下方）。

**查看详情**：点击表格「详情」按钮或双击行，右侧抽屉显示：
- **基本信息**：完整字段
- **统计图表**：领域分布柱状图 + 完整度饼图
- **关联图谱**：G6 力导向图，可拖拽节点

批量导入 JSON 格式示例：
```json
[
  {
    "tech_name_cn": "量子纠错",
    "tech_name_en": "Quantum Error Correction",
    "tech_domain": ["量子计算", "基础研究"],
    "current_status": "emerging",
    "tech_summary": "利用冗余量子比特纠正退相干误差的技术。"
  }
]
```

### 5.3 项目 / 机构 / 人员画像

操作与技术画像相同：搜索 → 列表 → 详情抽屉 → 批量导入。

字段因画像类型不同自动适配（详情抽屉显示所有返回字段）。

### 5.4 扫描监测

**触发扫描**：
1. 选择「分析时间范围」（起止日期）
2. 可选填入「技术领域」（支持多选）
3. 点击「开始扫描」

扫描为异步任务，约 2 秒后列表自动刷新。

**前沿技术列表**：展示扫描结果，点击「详情」查看所有维度评分（爆发 / 专利 / 引用 / 投资 / 政策 / 融合）。

**告警列表**：展示系统生成的监测告警，按级别（critical / warn / info）标色。

### 5.5 新技术发现

**触发发现扫描**：点击「触发发现扫描」，系统自动检测弱信号。

**弱信号列表**：展示检测到的弱信号，字段包括关键词、强度、新颖度、一致性。

**信号关联网络**：点击「网络图」，右侧抽屉用 G6 力导向图展示该信号关联的技术 / 机构 / 人员节点，可交互拖拽。

### 5.6 选题服务

**生成选题**：
1. 点击「生成选题」
2. 填写生成数量（1~100）
3. 可选填「分析周期」
4. 提交，约 3 秒后列表刷新

**状态筛选**：下拉框筛选 pending / accepted / rejected / revised。

**查看详情**：点击标题或「详情」按钮，右侧抽屉显示：
- 各维度评分进度条（热度 / 政策 / 影响力 / 去重 / LLM）
- 评审细项（新颖度 / 重要性 / 可行性 / 表达）
- 评审依据文本

**提交反馈**：抽屉右上角「提交反馈」按钮：
- 评审结论：接受 / 拒绝 / 修改
- 评分：1~10
- 评审意见：文字说明

## 6. API 接口速查

所有服务均有 `/docs` 接口文档（Swagger UI）：

| 服务 | 文档地址 |
|------|---------|
| 技术画像 | http://localhost:8001/docs |
| 项目画像 | http://localhost:8002/docs |
| 机构画像 | http://localhost:8003/docs |
| 人员画像 | http://localhost:8004/docs |
| 扫描监测 | http://localhost:8101/docs |
| 新技术发现 | http://localhost:8102/docs |
| 选题服务 | http://localhost:8103/docs |

### 常用接口示例

```bash
# 健康检查
curl http://localhost:8001/health

# 搜索技术画像
curl -X POST http://localhost:8001/api/v1/profile/tech/search \
  -H "Content-Type: application/json" \
  -d '{"keyword":"量子","page":1,"page_size":10}'

# 触发前沿技术扫描
curl -X POST http://localhost:8101/api/v1/frontier-tech/scan \
  -H "Content-Type: application/json" \
  -d '{"period_from":"2024-01-01","period_to":"2024-12-31","tech_domains":["人工智能"]}'

# 获取技术统计
curl http://localhost:8001/api/v1/stats/tech

# 生成选题
curl -X POST http://localhost:8103/api/v1/topics/generate \
  -H "Content-Type: application/json" \
  -d '{"target_count":5}'
```

## 8. 常见问题排查

### 服务一直处于 starting / unhealthy

```bash
# 查看该服务日志
docker compose logs --tail=50 profile_tech

# 查看所有服务状态
docker compose ps
```

常见原因：
- **elasticsearch 未就绪**：ES 启动需 60 秒，其他服务等待期间显示 `starting`，属正常。
- **migrate 失败**：查看 `docker compose logs migrate`，通常是数据库连接问题。
- **端口被占用**：本机 80 / 5432 / 9200 等端口有其他进程占用。

### 前端页面打开空白或 502

```bash
# 检查 nginx 配置是否生效
docker compose exec frontend nginx -t

# 检查对应后端是否 healthy
docker compose ps | grep profile_tech
```

### ES 启动后搜索无结果

搜索依赖 ES 索引，需先导入数据。索引在数据写入时自动创建。

### 清理并重新初始化数据库

```bash
# 停止所有服务并删除数据卷
docker compose down -v

# 重新启动（会重跑 migrate）
docker compose up -d --build
```

## 9. 数据管理

### 备份 PostgreSQL

```bash
docker compose exec postgres pg_dump -U metaprofile metaprofile > backup_$(date +%Y%m%d).sql
```

### 恢复 PostgreSQL

```bash
docker compose exec -T postgres psql -U metaprofile metaprofile < backup_20240101.sql
```

### 查看数据库

```bash
# 进入 psql
docker compose exec postgres psql -U metaprofile -d metaprofile

# 常用查询
\dt                          -- 列出所有表
SELECT count(*) FROM tech_profiles;
SELECT count(*) FROM frontier_tech_results;
SELECT count(*) FROM topic_candidates;
```

### 清理 ES 索引

```bash
# 查看所有索引
curl http://localhost:9200/_cat/indices?v

# 删除指定索引（谨慎）
curl -X DELETE http://localhost:9200/tech_profiles
```

## 10. 全量模式（LLM + Neo4j）

默认模式不启动 neo4j / rabbitmq / litellm（节省资源）。启用全量模式：

### 10.1 配置 LLM API Key

编辑 `deploy/.env`：

```env
QWEN_API_KEY=sk-你的阿里云DashScope密钥
DEEPSEEK_API_KEY=sk-你的DeepSeek密钥
LLM_PROXY_API_KEY=sk-自定义主密钥（任意字符串）
```

### 10.2 启动全量模式

```bash
docker compose --profile full up -d --build
```

新增启动的服务：

| 服务 | 端口 | 说明 |
|------|------|------|
| neo4j | 7474（Web）/ 7687（Bolt） | 图数据库，存储实体关系 |
| rabbitmq | 5672 / 15672（管理页） | 消息队列，异步任务 |
| litellm | 4000 | LLM 代理，统一调用大模型 |

### 10.3 访问 Neo4j 浏览器

打开 http://localhost:7474，用户名 `neo4j`，密码 `password`。

### 10.4 访问 RabbitMQ 管理页

打开 http://localhost:15672，用户名 `guest`，密码 `guest`。
